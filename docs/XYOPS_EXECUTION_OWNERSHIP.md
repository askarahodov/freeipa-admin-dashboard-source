# Ответственность за ограничения выполнения XYOps

Rate limits, очереди и ограничения параллельности не реализуются в портале. Они принадлежат XYOps как единственному оркестратору процессов.

## Почему ограничения находятся в XYOps

Ограничение только в web-интерфейсе можно обойти:

- прямым вызовом API XYOps;
- другим порталом;
- плагином;
- скриптом;
- вторым экземпляром приложения.

Поэтому XYOps должен атомарно решать:

- разрешён ли новый запуск;
- сколько Job одного процесса могут выполняться одновременно;
- ставится ли запрос в очередь;
- какой лимит действует для пользователя, группы или process ID;
- когда разрешён повторный запуск;
- какой `Retry-After` вернуть клиенту.

## Ответственность портала

Портал выполняет только UX-защиту и отображение решения XYOps:

- блокирует повторное нажатие кнопки на время HTTP-запроса;
- не создаёт собственную бизнес-очередь;
- передаёт пользователю HTTP `409` и `429`;
- показывает фиксированное безопасное объяснение причины;
- показывает `Retry-After`, когда заголовок присутствует;
- отображает текущие Job и их статусы.

Сырой текст и тело ошибочного ответа XYOps не возвращаются браузеру и не сохраняются в D1: upstream может содержать внутренние адреса, диагностические данные или секреты.

## Ожидаемые ответы XYOps

Конфликт параллельного запуска:

```text
HTTP 409 Conflict
```

Превышение частоты:

```text
HTTP 429 Too Many Requests
Retry-After: 120
```

Портал не должен автоматически повторять такой запрос. Пользователь повторяет запуск после указанного интервала или после освобождения concurrency slot.

## Безопасность и аудит

Портал записывает отказ запуска как `xyops.run` с outcome `failure`, HTTP-статусом и безопасным error code. Содержимое API key, request body и сырые ответы XYOps не сохраняются.

Approval не резервирует concurrency slot. После согласования XYOps повторно проверяет собственные ограничения во время фактического `run_event`. Это предотвращает устаревшие разрешения и гонки между несколькими клиентами.
