# Политики видимости каталога XYOps

Портал может ограничивать категории и отдельные Events/Workflows для portal identity,
групп и ролей. Фильтрация выполняется сервером, поэтому скрытый процесс нельзя
запустить прямым HTTP-запросом в обход интерфейса.

## Где применяются правила

Политика проверяется при:

- `GET /api/integrations/catalog`;
- получении динамических вариантов через `catalog/options`;
- `POST /api/integrations/catalog/run`;
- безопасном повторе завершённого запуска.

Полный снимок каталога продолжает храниться в D1 для синхронизации и обнаружения
изменений. Пользователь получает только разрешённую часть каталога.

## Формат

```json
{
  "version": 1,
  "defaultEffect": "allow",
  "adminBypass": true,
  "rules": [
    {
      "id": "hide-production-from-interns",
      "effect": "deny",
      "users": [],
      "groups": ["interns"],
      "roles": [],
      "categories": ["Production"],
      "processes": []
    },
    {
      "id": "allow-dba-backup",
      "effect": "allow",
      "users": [],
      "groups": ["dba"],
      "roles": [],
      "categories": [],
      "processes": ["database-backup"]
    }
  ]
}
```

`users`, `groups` и `roles` — селекторы субъекта. Между ними действует логика OR:
правило подходит, если совпал хотя бы один указанный пользователь, группа или роль.
Пустые селекторы субъекта означают всех пользователей.

`categories` и `processes` — селекторы ресурса. Между ними также действует OR.
Пустые селекторы ресурса означают весь каталог.

Если одновременно совпали `allow` и `deny`, побеждает `deny`. Если ни одно правило
не совпало, используется `defaultEffect`.

`adminBypass: true` позволяет администратору видеть и запускать весь каталог, чтобы
ошибочная политика не заблокировала управление. Для строгого применения правил к
администраторам установите `adminBypass: false`.

## Источники политики

Приоритет источников:

1. запись `current` в D1-таблице `catalog_visibility_policies`;
2. переменная `PORTAL_CATALOG_POLICIES_JSON`;
3. безопасная совместимая конфигурация по умолчанию:
   `defaultEffect=allow`, `adminBypass=true`, правил нет.

Администратор управляет политикой в разделе «Настройки → Видимость категорий и
процессов». API:

```text
GET /api/integrations/catalog/policies
PUT /api/integrations/catalog/policies
```

Оба запроса требуют роль `admin`/право `settings.manage` и корректный
`x-admin-token`. Сохранение требует D1.

## Доверенные группы

Клиентский заголовок `oai-authenticated-user-groups` всегда удаляется на security
boundary. Портал создаёт внутренний список групп только из доверенных источников:

- `PORTAL_STATIC_GROUPS` в static-режиме;
- `PORTAL_GROUPS_JSON` — централизованное отображение identity в группы;
- заголовок `PORTAL_GROUPS_HEADER` в proxy-режиме, но только после успешной проверки
  `PORTAL_PROXY_SHARED_SECRET`.

Пример отображения identity:

```env
PORTAL_GROUPS_JSON={"operator@company.local":["ops","linux"],"* ":["employees"]}
```

Для wildcard используйте ключ `"*"` без пробела:

```env
PORTAL_GROUPS_JSON={"operator@company.local":["ops","linux"],"*":["employees"]}
```

Пример proxy-конфигурации:

```env
PORTAL_GROUPS_HEADER=x-auth-request-groups
```

Reverse proxy должен удалить одноимённый заголовок клиента и сформировать новый
список групп после аутентификации. Значения разделяются запятыми или точками с
запятой.

## Ограничения

- до 200 правил;
- до 100 значений каждого вида в одном правиле;
- идентификаторы процессов сравниваются точно;
- названия категорий, групп и identity сравниваются без учёта регистра;
- неизвестные эффекты, дублирующиеся ID правил и некорректная структура отклоняются.

История операций и аудит имеют отдельное право `directory.read`. Политика каталога
контролирует обнаружение процесса и возможность нового запуска; она не удаляет уже
существующие аудиторские записи.
