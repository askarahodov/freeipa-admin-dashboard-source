# Автоматический acceptance-тест локальной аутентификации

Runner проверяет работающий локальный портал через реальные HTTP-запросы. Он не запускается в GitHub Actions и не обращается к FreeIPA или XYOps.

## Что проверяется

- доступность `/api/integrations/health`;
- вход существующего локального администратора;
- создание временных `viewer`, `operator` и `admin`;
- успешный вход каждой роли;
- запрет RBAC, diagnostics и session management для viewer/operator;
- доступ второго администратора к административным API;
- доступ всех аутентифицированных ролей к общему статусу портала;
- запрет понижения собственной активной admin-учётной записи;
- смена пароля и автоматический отзыв старой сессии;
- повторный вход новым паролем;
- отдельный отзыв выбранной сессии;
- автоматическое удаление созданных временных пользователей.

Пользователи получают уникальный префикс:

```text
portal-accept-<timestamp>-
```

## Запуск

Сначала запустите портал:

```bash
docker compose up -d --build
```

Создайте файл параметров:

```bash
cp .env.local-auth-acceptance.example .env.local-auth-acceptance
```

Заполните текущий логин и пароль локального администратора, затем выполните:

```bash
set -a
source .env.local-auth-acceptance
set +a
npm run test:local-auth:acceptance
```

## Защита от случайного запуска

Runner завершится до первого HTTP-запроса, если отсутствует:

```env
PORTAL_TEST_CONFIRM=YES
```

Также обязательны:

```env
PORTAL_TEST_ADMIN_USERNAME=<существующий администратор>
PORTAL_TEST_ADMIN_PASSWORD=<его текущий пароль>
```

Runner изменяет только внутреннюю базу пользователей портала. Он не создаёт и не удаляет пользователей FreeIPA и не запускает процессы XYOps.

## Отчёты

```text
artifacts/local-auth-acceptance/latest.json
artifacts/local-auth-acceptance/<run-id>/report.json
artifacts/local-auth-acceptance/<run-id>/report.html
```

В отчёт не записываются:

- пароль администратора;
- автоматически сгенерированные пароли тестовых пользователей;
- session cookies;
- session token hashes;
- заголовки Authorization;
- необработанные тела ошибок.

Известные секреты дополнительно заменяются на `[REDACTED]`.

## Cleanup

Cleanup выполняется в `finally`, даже если один из шагов завершился ошибкой. Каждый созданный пользователь удаляется в обратном порядке.

Если cleanup не смог удалить пользователя, это будет отмечено отдельным failed-шагом в отчёте. После такого результата найдите пользователей по префиксу `portal-accept-` в `/access` и удалите их вручную.

## Успешный результат

Команда возвращает exit code `0`, когда все проверки и cleanup прошли успешно. При ошибке возвращается exit code `1`, а ошибки конфигурации до запуска возвращают exit code `2`.
